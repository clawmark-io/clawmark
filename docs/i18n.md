# Internationalization (i18n)

All user-facing strings must be translated. Never hardcode display text in components.

## Stack

- **i18next** — core i18n framework
- **react-i18next** — React integration (`useTranslation` hook)
- **i18next-browser-languagedetector** — auto-detects language from localStorage / navigator
- **i18next-icu** — ICU message format for pluralization and formatting

## Supported Languages

- English (`en`) — fallback language
- Polish (`pl`)

## Translation Files

Translations live in `src/locales/{lang}/`, split into namespaces:

```
src/locales/
├── en/
│   ├── common.json      # buttons, labels, navigation, shortcuts
│   ├── commands.json     # command palette
│   ├── projects.json     # project management, templates
│   ├── settings.json     # appearance, workspace, language
│   ├── sync.json         # sync, import/export
│   └── tasks.json        # task UI, dates, filtering, sorting
└── pl/
    └── (same files)
```

The default namespace is `common`.

## Configuration

`src/i18n.ts` initializes i18next with:

- All language resources imported statically
- Fallback language: `en`
- Detection order: localStorage key `app-language`, then navigator
- ICU plugin enabled for plural rules and number/date formatting

## Usage in Components

```tsx
import { useTranslation } from "react-i18next";

function MyComponent() {
  const { t } = useTranslation("tasks"); // namespace
  return <h1>{t("title")}</h1>;
}
```

For language-aware date/number formatting, use `i18n.language` with `Intl` APIs:

```tsx
const { i18n } = useTranslation();
new Intl.DateTimeFormat(i18n.language).format(date);
```

## ICU Features

Pluralization uses ICU syntax, which supports language-specific plural rules:

```json
"taskCount": "{count, plural, one {# task} other {# tasks}}"
```

Polish has additional plural forms (`few`, `many`) that must be provided.

## Type-Safe Translation Keys

All `t()` calls are type-checked at compile time. Passing a key that does not exist in the namespace JSON file is a TypeScript error.

The type declaration lives in `src/types/i18next.d.ts`. It augments i18next's `CustomTypeOptions` interface, mapping each namespace name to the shape of its English JSON file. The English JSON files are the single source of truth for valid keys.

## Adding a New Translation Key

1. Add the key to the appropriate namespace file in `src/locales/en/`.
2. Add the corresponding translation in `src/locales/pl/`.
3. Use `t("key")` with the correct namespace in the component.
4. Run `npm run build` to verify the key exists and the type system accepts it.

## Language Switching

The language selector is in Settings. Selection is persisted to localStorage (`app-language`) and applied via `i18n.changeLanguage()`.
