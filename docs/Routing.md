# Routing

The app uses [TanStack Router](https://tanstack.com/router) with file-based routing. Routes are defined as files under `src/routes/` and a route tree is auto-generated by the Vite plugin.

## Setup

- **Hash history** — `createHashHistory()` is used because Tauri loads the app via `file://` protocol, which doesn't support HTML5 push-state routing.
- **Router context** — `WorkspacesManager` is created in `main.tsx` and passed to the router via context. The root route (`__root.tsx`) reads it to provide `ManagerProvider`.
- **Vite plugin** — `@tanstack/router-plugin/vite` (`TanStackRouterVite`) auto-generates `src/routeTree.gen.ts` from the `src/routes/` directory.

## Route Structure

```
src/routes/
  __root.tsx                          # Global providers, hooks, command palette
  index.tsx                           # / -> redirect to /w
  w/
    index.tsx                         # /w -> workspace picker
    $workspaceId/
      route.tsx                       # Layout: WorkspaceProvider, sidebar, shortcuts, dialogs
      index.tsx                       # Redirect to /projects or /upcoming based on defaultView setting
      projects.tsx                    # Projects view (project grid)
      upcoming.tsx                    # Upcoming view
      settings.tsx                    # Workspace settings (?tab=general|look-and-feel|cloud-sync)
      sync.tsx                        # Sync view (?tab=sync|import|export)
      p/
        $projectId/
          route.tsx                   # Layout: project validation, background, header
          index.tsx                   # Redirect to kanban (or tasks if kanban disabled)
          kanban/
            route.tsx                 # Layout: KanbanView + Outlet
            index.tsx                 # (empty, kanban rendered by layout)
            $taskId.tsx               # Task detail dialog overlay
          tasks/
            route.tsx                 # Layout: TasksView + Outlet
            index.tsx                 # (empty, tasks rendered by layout)
            $taskId.tsx               # Task detail dialog overlay
          settings.tsx                # Project settings
          workspace-settings.tsx      # Workspace settings (in project context)
          sync.tsx                    # Sync view (in project context)
```

## Key Patterns

### Layout Routes

Files named `route.tsx` act as layouts. They render shared UI (sidebar, header, background) and an `<Outlet />` for child content. This keeps parent views mounted when navigating between children.

### Task Detail as Nested Route

Task detail is a dialog rendered by `$taskId.tsx` child routes under `kanban/` and `tasks/`. The parent layout (`kanban/route.tsx` or `tasks/route.tsx`) renders the board/list view and stays mounted while the dialog is open. Closing the dialog navigates back to the parent.

### Search Params for Tabs

Settings and sync views use validated search params for tab selection:

```tsx
export const Route = createFileRoute("/w/$workspaceId/settings")({
  validateSearch: z.object({
    tab: z.enum(["general", "look-and-feel", "cloud-sync"]).optional(),
  }),
});
```

### Settings/Sync in Project Context

Workspace settings and sync can be accessed without leaving the project context. Routes like `/w/:id/p/:pid/workspace-settings` and `/w/:id/p/:pid/sync` reuse the same `SettingsView` and `SyncView` components, wrapped in the project layout.

### Accessing Params

Use `useParams({ strict: false })` to access route params at any nesting level without coupling to a specific route.

## Navigation

All navigation uses TanStack Router's `useNavigate()` hook:

```tsx
const navigate = useNavigate();
navigate({ to: "/w/$workspaceId/p/$projectId/kanban", params: { workspaceId, projectId } });
```

The command palette and keyboard shortcuts also use `navigate()` via `CommandContext`.

## E2E Routing Test

`e2e/routing.spec.ts` navigates through every route in the application to verify they load correctly. It creates a test workspace via `window.debug.createTestWorkspace()`, then visits each route by direct URL and by clicking through the sidebar/UI.

**When adding a new route, add a corresponding navigation step to this test.** The test covers:

- Root redirect (`/` → `/w`)
- Workspace picker (`/w`)
- Default view redirect (`/w/:id` → `/w/:id/projects` or `/w/:id/upcoming` based on workspace setting)
- Projects / project grid (`/w/:id/projects`)
- Upcoming, Settings (all tabs), Sync (`/w/:id/upcoming`, `/w/:id/settings`, `/w/:id/sync`)
- Project redirect, Kanban, Tasks (`/w/:id/p/:pid`, `.../kanban`, `.../tasks`)
- Task detail dialogs (`.../kanban/:taskId`, `.../tasks/:taskId`)
- Project settings, workspace-settings in project context, sync in project context
- Full click-through navigation via sidebar buttons

## E2E Back/Forward Test

`e2e/routing-back-forward.spec.ts` verifies that browser back/forward navigation works correctly for every route. Each route has an individual test that:

1. Navigates to a source route that can reach the tested route
2. Clicks an element to navigate to the tested route
3. Invokes browser back navigation, waits, and asserts the URL is correct
4. Invokes browser forward navigation, waits, and asserts the URL is correct

**Every route must have a corresponding back/forward test.** When adding a new route, add a test case to this file alongside the navigation test in `routing.spec.ts`.
